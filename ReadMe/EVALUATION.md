# 记忆 + 知识库系统 v3 — 全面评价

> 评价日期：2026-05-27 | 评价人：Hermes Reviewer | 代码量：~1200 行（6 个文件）

---

## 一、一句话定位

**一个确定性优先、手动可控的 Agent 记忆持久化层。** 不是通用知识库，不是 RAG 引擎——它是 AI Agent 决策/结论/知识的副作用持久化系统。

---

## 二、架构评分

### R1 存储层 — ⭐⭐⭐⭐½

**优势：**
- 三级存储（L0 索引 / L1 摘要 / L2 全文）设计合理，按需加载不浪费 token
- 三库隔离（全局记忆 / 知识库 / Profile 专属），职责清晰
- 字段完备：`version`/`status`/`superseded_by_id`/`topic`/`relation_to_prev` + 2560 维向量

**不足：**
- `memory_meta` 和 `knowledge_meta` 结构不统一（历史原因），查询需两套 JOIN
- SQLite 上限在百万级，不适用超大规模部署

### R2 注入层 — ⭐⭐⭐⭐

**优势：**
- 两级压缩（DeepSeek → LM Studio），质量远高于单级 LLM 抽取
- 结构化提取稳定（13 个字段 + L0/L1/决策），解析器正则健壮
- 手动触发是设计选择，不是缺陷——避开了 Hermes 自动钩子不可靠的坑

**不足：**
- 注入耗时 40-60s（两级 LLM 调用），不适合高频场景
- 空 topic 时不触发版本覆盖（兼容旧数据，但新数据若 LLM 漏输出 topic 也不会覆盖）

### R3 检索层 — ⭐⭐⭐⭐½

**优势：**
- 三层互补：FTS5（精确） + 语义搜索（模糊） + 结构化过滤（定向）
- `--include-obsolete` 可查看完整版本链
- `--rerank` 用 LLM 做语义重排序，弥补向量相似度的语义盲区
- 前端展示版本号 + 状态 + 相似度分数

**不足：**
- Rerank 只基于 abstract 摘要（<20 字），不看完整 L1，信息量有限
- `--rerank` 与 `--hybrid` 组合使用体验还未验证

### R4 版本管理层 — ⭐⭐⭐⭐

**优势：**
- 二击保险是亮点：第一次 `downgraded` 仍可见，第二次才真正 `superseded`
- LLM 误判只降级不删数据，两次误判概率极低
- 版本链可追溯，`superseded_by_id` 可跳转到新版本

**不足：**
- `downgraded` 记录在搜索结果中没有视觉标识（用户不知道"这条即将被覆盖"）
- 知识库的 `supplements`/`independent` 关系不影响搜索排序——补充内容应该排在原内容之后，目前没有实现

### R5 图谱推理层 — ⭐⭐⭐

**优势：**
- 传递闭包（2 跳 BFS）轻量无依赖，纯 SQL 操作
- 推理结果标记 `inferred`，区分手动关联和自动推理
- 知识库和记忆库都支持

**不足：**
- 缺乏实体抽取——当前只能通过"关联:slug"和 tag 重叠建立链接，不能从文本中自动发现实体关系
- 图谱推理只有"关联"，没有"因果"和"时序"维度
- `inferred` 链接不参与搜索排序，用户看不到推理链

### R6 运维层 — ⭐⭐⭐⭐⭐

**优势：**
- 零外部依赖，6 个 Python 文件 + SQLite
- 全手动触发，无 daemon/service crash 风险
- 所有数据可脱离系统直接用 SQLite 查询

## 四、已知问题

### P2 — 注入流程中 knowledge_ids 可能为空导致跳过推理

```python
if knowledge_ids:  # 如果 soft_dedup 全部 merge，knowledge_ids 可能为空
    _infer_transitive_links(kdb2, knowledge_ids)
```

`write_knowledge_l0_l1()` 的 `knowledge_ids` 在全部 soft_dedup merge 时可能为空，此时传递闭包不会执行。影响很小——merge 意味着内容几乎一样，不推理也不会丢失链接。

### P2 — `rerank()` 的 prompt 只传 abstract，L1 摘要更丰富

当前 rerank 只传 abstract（<20 字），LLM 获得的信息有限。改成传 L1 摘要（~200 字）会大幅提升 rerank 质量，但 token 成本增加。

### P3 — SQLite 并发写限制

多个注入同时执行时会写锁串行。当前单用户场景不存在此问题。

## 五、最佳适用场景

| 适合 | 不适合 |
|---|---|
| 个人/团队 AI Agent 的决策记忆 | 百万级文档检索 |
| 增量式知识积累（注入→覆盖→推理） | 高频实时检索 |
| 需要版本管理和审计的知识场景 | 多用户协作（无权限模型） |
| 低成本、零运维的知识持久化 | 需要完整图谱推理 |

## 六、改进优先级

| 优先级 | 改进项 | 难度 |
|---|---|---|
| P1 | 搜索结果中标记 `downgraded` 记录（视觉提示即将被覆盖） | 低 |

| P2 | LLM 提取"因果/时序"关系维度的实体 | 中 |
